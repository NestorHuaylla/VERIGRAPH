REPORTE FINAL DE EJECUCION LOCAL - 2026-06-02

Motivo
- El usuario queria ver todo el desarrollo corriendo en navegador.
- El enlace esperado era:
  - `http://127.0.0.1:3000`
- Al inicio salia error tipo:
  - `failed to load page`
  - no cargaba `localhost:3000` / `127.0.0.1:3000`.

Diagnostico realizado
- Se reviso estado de Docker con:
  - `docker-compose ps`
- Resultado observado:
  - solo estaban levantados servicios de infraestructura:
    - `verigraph-postgres-1`
    - `verigraph-redis-1`
    - `verigraph-neo4j-1`
    - `verigraph-minio-1`
  - no estaban levantados:
    - `web`
    - `api`
    - `worker-external`
    - `worker-analysis`
    - `worker-alerts`
    - `keycloak`
- Se revisaron puertos con `netstat`.
- Puertos vivos por Docker:
  - PostgreSQL: `5432`
  - Redis: `6379`
  - Neo4j Browser: `7474`
  - Neo4j Bolt: `7687`
  - MinIO API: `9000`
  - MinIO console: `9001`
- Puertos que faltaban al inicio:
  - Web: `3000`
  - API: `8000`

Problema Docker detectado
- En esta maquina `docker compose` no funciona como plugin moderno.
- El comando valido es:
  - `docker-compose`
- Ademas, Docker Desktop reporto error interno de WSL al intentar inspeccionar/levantar servicios:
  - `Docker Desktop is unable to start`
  - `exit status 139`
  - errores relacionados a `docker-desktop`, `wsl-bootstrap` y `backend.sock`.
- Tambien aparecio:
  - `WARNING: Error loading config file: open C:\Users\PC\.docker\config.json: Acceso denegado.`
  - `permission denied while trying to connect to the docker API at npipe:////./pipe/docker_engine`
- Conclusion:
  - Docker Desktop/WSL tiene un problema interno de permisos/arranque.
  - La infraestructura publicada por Docker aparecia con puertos abiertos, pero no era confiable para levantar todo con build Docker.
  - Construir imagenes `web/api/workers` se quedaba trabado demasiado tiempo.

Solucion aplicada para que el usuario pueda ver el desarrollo
- Se evito reconstruir imagenes Docker.
- Se uso Docker solo como infraestructura ya levantada:
  - Postgres.
  - Redis.
  - Neo4j.
  - MinIO.
- Se levanto la API y la web como procesos locales de Windows.
- Esto es mas rapido y estable para desarrollo local que esperar builds Docker largos.

Incidente produccion 2026-06-04 - registro/login fallaba

Sintomas observados en servidor
- El registro/login devolvia `500 Internal Server Error`.
- En logs del API aparecio primero:
  - `asyncpg.exceptions.InvalidPasswordError: password authentication failed for user "verigraph"`.
- Tambien aparecian warnings al ejecutar Compose sin env file:
  - `The "POSTGRES_PASSWORD" variable is not set. Defaulting to a blank string.`
  - `The "POSTGRES_USER" variable is not set. Defaulting to a blank string.`
- Luego se reviso un traceback con:
  - `UndefinedTableError: relation "users" does not exist`.
  - Se valido la lista de tablas y finalmente `users` si existia; ese punto fue un error de lectura/copiado en terminal.

Causa real
- Docker Compose se estaba ejecutando sin `--env-file .env.production`, por eso las variables usadas en interpolacion de `docker-compose.prod.yml` salian vacias en los warnings.
- Ademas, el volumen de PostgreSQL ya estaba inicializado con una password interna distinta a la indicada en `.env.production`.
- Cambiar `POSTGRES_PASSWORD` en `.env.production` no cambia automaticamente la password real de un usuario ya creado dentro del volumen de Postgres.
- El API usa `DATABASE_URL=postgresql+asyncpg://verigraph:poni4545@postgres:5432/verigraph`; si la password real del usuario `verigraph` no coincide, falla cualquier consulta, incluido registro/login.

Solucion aplicada
- Ejecutar Docker Compose siempre con env file:
```bash
sudo docker compose --env-file .env.production -f docker-compose.prod.yml ps
sudo docker compose --env-file .env.production -f docker-compose.prod.yml up -d
```
- Sincronizar la password real del usuario de Postgres con la de `.env.production`:
```sql
ALTER USER verigraph WITH PASSWORD 'poni4545';
```
- Reiniciar el API:
```bash
sudo docker compose --env-file .env.production -f docker-compose.prod.yml restart api
```

Validaciones que confirmaron la reparacion
- Conexion a Postgres:
```bash
sudo docker compose --env-file .env.production -f docker-compose.prod.yml exec postgres psql -U verigraph -d verigraph -c "SELECT current_user;"
```
- Resultado esperado:
```text
 current_user
--------------
 verigraph
```
- Tablas existentes:
```bash
sudo docker compose --env-file .env.production -f docker-compose.prod.yml exec postgres psql -U verigraph -d verigraph -c "SELECT tablename FROM pg_tables WHERE schemaname='public' ORDER BY tablename;"
```
- Deben aparecer, entre otras:
  - `users`
  - `reports`
  - `risk_rules`
  - `risk_scores`
- Prueba de API:
```bash
curl -i http://127.0.0.1:8000/api/v1/health
curl -i -X POST "http://127.0.0.1:8000/api/v1/auth/login" -H "Content-Type: application/json" --data-raw '{"email":"huallbenit@gmail.com","password":"poni4545"}'
```

Nota importante
- El problema no fue normalizacion del correo ni CORS.
- El problema principal fue inconsistencia entre `.env.production`, Compose y la password real guardada en el volumen de PostgreSQL.
- En produccion se recomienda cambiar `poni4545` por secretos largos y distintos para `POSTGRES_PASSWORD`, `DATABASE_URL`, `JWT_SECRET`, `NEXTAUTH_SECRET`, `WORKER_API_TOKEN`, `NEO4J_PASSWORD`, `MINIO_ROOT_PASSWORD` y `S3_SECRET_KEY`.
